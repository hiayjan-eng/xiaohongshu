# 收藏复活系统产品蓝图

## 1. 当前状态

当前项目是一个已经上线的 Web MVP，线上地址为 `https://xiaohongshu-green.vercel.app`。仓库采用 pnpm monorepo，主要包含：

- `apps/web`：React/Vite Web 工作台，承担导入、收藏池、搜索、智能专辑、今日复活、设置、QA 和真实试用。
- `apps/extension`：Chrome / Edge Manifest V3 浏览器扩展 Beta，用于用户主动在本人小红书网页版页面上扫描已加载 DOM。
- `apps/mobile`：移动端分享入口技术预留，目前不是完整原生 App，也不是 WebView App。
- `packages/*`：共享类型、导入管线、分类服务、AI Provider、搜索、推荐、行动卡、数据库规范化和存储适配。
- `api/ai.ts`：Vercel serverless AI 代理，真实 Key 只允许在服务端读取；无 Key 时 fallback 到 mock。

当前数据仍保存在当前浏览器的 `localStorage`，生产环境没有登录、云同步、数据库、真实小红书接口，也没有正式商店版扩展。

## 2. 产品定位

收藏复活系统是一个个人收藏行动助手。它不做内容社区，不重建小红书内容库，也不把收藏夹变成更复杂的知识库。它的核心价值是：

> 把散落、吃灰的收藏重新找回来，整理清楚，需要时再转成一个可以完成的小动作。

产品的默认动作不是“把所有收藏变成任务”，而是先建立可搜索、可整理、可纠正的收藏索引。只有当用户选择“复活这条”时，系统才按需生成行动卡；只有当用户主动加入计划时，才生成轻量 PlanCard。

## 3. 目标用户

第一批目标用户是小红书重度收藏者，尤其是：

- 内容创作者：收藏封面、选题、脚本、运营案例，需要快速找回和改写成自己的选题。
- 学习型用户：收藏 AI 工具、技能教程、英语、摄影、设计、写作等内容，但不愿意每条都手动整理。
- 生活方式用户：收藏旅行路线、探店、菜谱、家居改造、穿搭和购物参考，需要在某个场景里找回。
- 职场和商业用户：收藏招聘、创业团队、商业案例、工作流、决策方法，希望按主题和用途重新归档。
- 情绪和关系记录者：收藏观点、关系表达、自我观察、手帐素材，不希望被误判成内容创作素材。

## 4. 用户问题

当前真实问题不只是“收藏太多”，而是收藏进入了一个不可行动、不可回忆、不可搜索的状态：

- 用户过去收藏了几百到几千条内容，逐条分享导入成本太高。
- 用户记得大概主题，却不记得标题、作者或路径。
- 收藏内容有两层含义：它讲的主题是什么，以及用户为什么收藏它。当前任何产品如果把两者混在一起，都会造成误分类。
- 用户不希望导入后被强行塞满待办，也不希望看到三天、七天计划自动生成。
- 浏览器扩展、Web 和未来手机 App 的职责如果不清楚，后续会在每个平台上重复造功能。

## 5. 产品原则

1. 导入先生成收藏索引，不自动制造大量 ActionCard。
2. 内容主题 `contentDomain/contentSubDomain` 和收藏用途 `savedIntent` 必须分离。
3. AI 只做建议，用户纠正结果优先于自动分类。
4. 用户手动移出、手动加入、手动改名等操作不得被后续自动整理覆盖。
5. 批量变更必须可预览、可备份、可撤销。
6. 搜索的目标是找回原帖和自己的收藏索引，不是展示原帖全文。
7. 浏览器扩展只在用户本机、本人登录、主动点击后读取当前页面已加载 DOM。
8. 不登录小红书账号，不做云端爬虫，不模拟登录，不绕过验证码或风控。
9. 手机 App 负责系统分享导入、搜索找回和日常复活，不承诺直接读取完整小红书收藏夹。
10. 数据迁移必须优先保护用户已有收藏，不自动清空 localStorage，不自动覆盖分类纠正。

## 6. 北极星指标

北极星指标：每周被成功找回并完成一个最小动作的收藏数。

辅助指标：

- 扫描到导入转化率：扫描结果中被用户确认导入的比例。
- 搜索找回率：用户搜索后点击“查看收藏”或“打开原帖”的比例。
- 复活率：SavedItem 被生成 ActionCard 的比例。
- 计划率：ActionCard 被用户主动加入 PlanCard 的比例。
- 完成率：PlanCard 被标记完成的比例。
- 分类纠正率：用户纠正主题或用途的比例，越高说明 AI 建议需要优化。
- 高置信误判率：高置信分类被纠正的比例，必须长期保持低位。

## 7. 平台分工

### Web 工作台

Web 是批量整理和深度管理中心。它负责：

- 旧收藏扫描结果接收和 ImportBatch 管理。
- 收藏池、搜索、智能专辑、分类纠正、文本迁移、数据备份。
- 按需生成 ActionCard 和 PlanCard。
- 设置、主题、AI 状态、开发与测试入口。

当前状态：Web 已是主要可用产品，但 `apps/web/src/App.tsx` 过于集中，后续要拆页面和状态层。

### 浏览器扩展

扩展负责旧收藏扫描。它只做用户本机、当前页面、可见或滚动加载后 DOM 的读取。

当前状态：`apps/extension` 已有 Manifest V3 Beta、扫描、自动滚动、暂停继续、进度条、断点、导入 Web Bridge 和 ZIP 构建产物。它不是商店版，也不应承诺“云端一键读取完整收藏夹”。

### 手机 App

手机 App 的第一价值是低摩擦新收藏导入和日常复活，不是批量扫描历史收藏夹。

目标能力：

- Android `ACTION_SEND` / `ACTION_SEND_MULTIPLE` 接收链接和文本。
- iOS Share Extension 接收 URL/text，并通过 App Group 交给主 App。
- 手机端搜索、今日复活、收藏详情、行动卡和轻量计划。
- 和 Web 共享云端数据后，手机和 Web 看到同一套收藏索引。

明确限制：手机 App 无法安全、合规地直接读取用户完整小红书收藏夹。历史收藏批量整理应主要依赖桌面浏览器扩展。

## 8. 全部用户流程

### 旧收藏扫描流程

1. 用户打开 Web `/old-import`，下载扩展 Beta。
2. 用户在 Chrome / Edge 安装 unpacked extension。
3. 用户打开本人小红书网页版收藏页面。
4. 用户主动点击扩展扫描当前已加载收藏。
5. 用户可自动加载更多、暂停、继续、筛选、取消部分结果。
6. 用户点击导入 Web。
7. Web 创建 ImportBatch 和 SavedItem，仅生成收藏索引和智能专辑候选。
8. 用户在智能专辑或收藏池中选择某条收藏“复活这条”。
9. 系统按需生成 ActionCard。
10. 用户主动加入今日计划，形成 PlanCard。

### 新收藏导入流程

1. 用户复制小红书分享链接或整段分享文本。
2. Web `/import` 或未来手机 App 分享入口解析 URL、标题、分享文案和备注。
3. ImportBatch 去重后创建 SavedItem。
4. 系统生成主题、用途、摘要、关键词、搜索索引和智能专辑候选。
5. 用户可继续导入、搜索、查看专辑或复活单条。

### 搜索找回流程

1. 用户输入地点、技能、菜名、工具、主题或模糊描述。
2. 搜索匹配 displayTitle、cleanedTitle、rawTitle、userNote、visibleText、keywords、entities、contentDomain、contentSubDomain、savedIntent 和 SmartAlbum title。
3. 结果展示匹配原因。
4. 用户选择打开原帖、查看收藏或复活这条。

### 智能专辑流程

1. 系统按主题视角和用途视角生成候选。
2. 候选不自动长期收纳未来收藏。
3. 用户确认后，专辑变成 confirmed，并选择高度匹配自动加入、中等匹配待确认。
4. 用户可归档候选，归档不删除任何收藏。
5. 用户可在详情页手动移动主题、添加用途专辑、移出收藏。

### 计划流程

1. SavedItem 先被复活成 ActionCard。
2. 用户确认加入计划。
3. PlanCard 出现在今日复活。
4. 用户可开始、完成、延期、改日期、取消、查看来源收藏。
5. 系统不自动生成多日任务。

## 9. 信息架构

主导航建议：

- 今日复活：日常入口，包含今日计划和收藏搜索。
- 导入中心：旧收藏扫描为主入口，新收藏手动导入为次入口。
- 智能专辑：主题视角、用途视角、待确认新增和已归档。
- 搜索：找回原帖和收藏索引。
- 收藏池：所有 SavedItem 的管理视图。
- 设置：主题、AI 状态、本地数据、文本迁移、开发与测试入口。

降级或隐藏：

- 计划库从主导航降级为今日复活和收藏详情内的能力。
- QA 和真实试用默认不出现在 production 普通侧边栏，只在设置的“开发与测试”里打开。

## 10. 页面列表

当前已经存在或目标保留的页面：

- `/`：Welcome / Landing。
- `/dashboard`：今日复活。
- `/import`：导入中心。
- `/old-import`：旧收藏扫描 Beta 安装和状态页。
- `/albums`：智能专辑列表。
- `/albums/:albumId`：智能专辑详情。
- `/search`：全局搜索。
- `/pool`：收藏池。
- `/detail` 或内部详情状态：收藏详情和行动卡详情。
- `/settings`：设置。
- `/qa`：开发 QA。
- `/real-test`：真实试用模式。

未来新增：

- 手机 App：今日、搜索、收藏、专辑、我的。
- 云同步后：登录、迁移确认、设备管理、数据导出。

## 11. 功能优先级

P0：

- IndexedDB 数据底座。
- 扫描 0 新增状态解释。
- 今日复活搜索和布局稳定。
- 智能专辑生命周期语义收口。
- 文本迁移预览、备份、应用、撤销。

P1：

- 云同步和登录。
- 手机 App MVP 分享入口。
- AI Provider 真实生成和质量闭环。
- 扩展 Beta 产品化和安装体验。

P2：

- 正式扩展商店发布。
- 多端同步冲突处理。
- 个性化推荐和复盘。
- 手帐、对话式收藏整理和高级分析。

## 12. 非目标

- 不做小红书账号登录。
- 不读取用户云端收藏夹。
- 不做云端爬虫或公开内容数据库。
- 不复制、保存或展示原帖全文。
- 不绕过验证码、登录墙、风控或平台限制。
- 不在导入时自动制造大量任务。
- 不把手机 App 做成简单 WebView 并宣称原生能力完成。

## 13. MVP、V1、V2 边界

MVP 边界：Web + unpacked 扩展 Beta + localStorage。核心闭环是扫描/导入、收藏索引、搜索、智能专辑候选、按需复活、轻量 PlanCard。

V1 边界：IndexedDB、稳定扫描状态、可回滚迁移、智能专辑生命周期完整、朋友测试流程稳定。仍可不登录。

V2 边界：Supabase 登录与云同步、手机 App MVP、真实 AI Provider、扩展 Beta 更接近商店版。所有外部凭证必须由用户提供后再实现。
